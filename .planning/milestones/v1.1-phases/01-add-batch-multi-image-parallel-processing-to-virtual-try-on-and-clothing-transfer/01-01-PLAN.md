# Plan 01-01: Batch Multi-Image Parallel Processing for Virtual Try-On and Clothing Transfer

## Goal

Ship bounded-parallel batch processing for:
1. Virtual Try-On: many subject images + one shared outfit set
2. Clothing Transfer: many concept images + one shared reference outfit set

while preserving the current single-item UX path, existing provider routing, and existing per-request `numberOfImages` behavior.

## must_haves

- [x] User can upload multiple subject images in Virtual Try-On
- [x] User can upload multiple concept images in Clothing Transfer
- [x] One shared 1-2 image outfit/reference set applies to every batch item in the run
- [x] Batch items run in parallel with a bounded worker pool, not strict serial execution
- [x] Per-item status is visible: pending, processing, completed, error
- [x] One failed batch item does not kill the whole run
- [x] Single-item use still works without forcing a different workflow
- [x] New i18n strings exist in both locales
- [x] Build, lint, and targeted tests all pass

<action>
Add explicit batch item contracts in `types.ts` for Virtual Try-On and Clothing Transfer, including item identity, source image, status, results, and error text.

Rewrite `hooks/useVirtualTryOn.ts` to become the real feature hook used by the component. It must:
- own `subjectImages` instead of a single `subjectImage`
- keep current clothing-item set, prompts, image count, aspect ratio, resolution, generated state, and upscale state
- expose bounded-parallel batch execution that constructs one try-on request per subject image using the current shared outfit set
- preserve existing Gemini vs local/anti request assembly by reusing the prompt builder logic, not redefining provider contracts

Extend `hooks/useClothingTransfer.ts` similarly so it owns `conceptImages` instead of one `conceptImage`, uses one request per concept image, preserves concept-first image ordering in each request, and tracks per-item statuses/results/errors.

Use a worker-pool pattern inspired by `useWatermarkRemover.ts` with a fixed internal concurrency cap. Do not add a user-facing concurrency control in this phase.
</action>
